# Compass ボール受け渡し操作計画

Last updated: 2026-03-06

## 前提

Compass のボールは「タスク操作」ではなく、「受け渡し」を雑に回せることが主目的。

そのため、主操作はボタンではなくスワイプに寄せる。
ただしスワイプを重くしすぎないことが前提で、以下を守る。

- 1回のスワイプで既定の相手に投げられる
- ミスしてもすぐ undo できる
- メモや催促日は必要な時だけ添えられる
- 誰に投げるかを毎回考えさせない

## なぜボタン中心にしないか

建築の受け渡しは回数が多い。

毎回、

- 相手を選ぶ
- メモを書く
- 日付を決める
- 確認ボタンを押す

だと、すぐに重くなる。

Compass では「まず回す」を優先し、
詳細はあとから足せるようにする。

## 想定シーン

### 1. 自分から相手に返す

- 図面差し替えを反映したのでクライアント確認に回す
- 見積条件を整理したので設備業者に返す
- 施工からの質問に答えたので現場へ返す

必要なのは:

- 既定の相手にすぐ投げる
- 必要なら一言を添える
- 何日返答がなければ催促するかを残す

### 2. 相手待ちを追う

- 先方承認待ち
- 協力会社の回答待ち
- 行政・審査機関の返答待ち

必要なのは:

- 今どの相手待ちか分かる
- 催促日が来たら浮いてくる
- 必要なら自分に引き戻せる

### 3. 差し戻しや引き取り

- 返答が遅いので自分で再対応する
- 条件が変わったのでいったん引き取る
- 投げ先が違ったので即戻す

必要なのは:

- 素早く自分に戻せる
- 戻した理由を一言だけ残せる
- 操作の undo がすぐできる

## 提案する操作モデル

### 主操作

#### 自分ボール一覧

- 右スワイプ: 既定の相手に投げる
- 左スワイプ: 今は未使用か、後で「完了」に使う

まずは右スワイプだけを強くする。
左右に役割を増やしすぎると、逆に怖くなるため。

#### 相手待ち一覧

- 左スワイプ: 自分に引き取る
- 右スワイプ: Phase 2 で「催促する」に拡張候補

最初の段階では、

- 自分ボールで「投げる」
- 相手待ちで「受け取る」

の2つを安定させるのが先。

## スワイプ後の軽い補助UI

スワイプは即時実行する。
その代わり、実行直後に画面下から短い補助シートを出す。

入れる項目は 3 つだけ。

- 一言メモ
- 催促日
- 戻す

これなら、

- 雑に使う人はそのまま終われる
- 必要な時だけ一言と催促日を足せる

## メモの考え方

メモはチャットにしない。

Compass で必要なのは会話ログではなく、受け渡し時点の短い文脈。

MVP では以下で十分。

- 最新の一言メモを 1 件持つ
- 詳細な履歴は既存の activity log に残す

候補文のプリセットを用意すると良い。

- 確認お願いします
- 差し替え済みです
- 金額確認お願いします
- 承認ください
- 先方回答待ちです

## 催促日の考え方

`responseDeadline` と `催促日` は分けるべき。

- `responseDeadline`: 理想的にはいつまでに返してほしいか
- `催促日`: 返答がなくても自分が確認しに行く日

建築実務では、

- 回答期限は明確に切れない
- でも 2 日後に状況確認したい

という場面が多い。

そのため、新しく内部項目を追加する。

- `ballFollowUpOn` または `ballNudgeOn`

UI では `催促日` として表示する。

## 誰に投げるか問題への対処

「毎回誰に投げるかを考えない」を成立させるには、既定の返し先が必要。

既存の `assignee` だけでは足りない場面がある。

例:

- 実務担当は設計者
- でも今返したい相手はクライアント

なので、MVP では以下を追加する。

- `ballDefaultRecipient`

決め方はこの順。

1. 明示設定された既定相手
2. 直前のやり取り相手
3. assignee

最初の 1 回だけ手動で選ばせ、以降は覚える形がよい。

## データモデル案

Task に最低限追加したいのは以下。

- `ballDefaultRecipient?: string | null`
- `ballFollowUpOn?: string | null`
- `ballLastActionAt?: string | null`
- `ballLastActionBy?: string | null`

既存項目はそのまま活かす。

- `ballHolder`
- `ballNote`
- `responseDeadline`

履歴は新しい独自配列をまず持たず、既存の activity log を利用する。

## 画面反映

### スマホ

- 自分ボール一覧
- 相手待ち一覧
- 今日催促一覧

カードには最低限だけ出す。

- PJ 名
- 節目または工程名
- 今の一言メモ
- 催促日 or 返答期限
- 今誰ボールか

### 工程画面

各行に小さく出す。

- 今誰ボール
- 催促日
- 期限超過

これで「受け渡し」と「工程」を切らない。

## 実装順

### Phase 1

- 右スワイプで投げる
- 左スワイプで引き取る
- undo
- `ballFollowUpOn` の追加
- スワイプ直後の補助シート

### Phase 2

- 催促日が来たカードを自動で上に出す
- プリセットメモ
- 既定の返し先を記憶する

### Phase 3

- 相手待ち一覧で右スワイプ催促
- 通知やリマインド連携
- 工程画面内のボール密度を上げる

## いまの判断

方向性としてはかなり良い。

特に重要なのはこの 3 点。

- 主操作をスワイプにする
- メモは必須にしない
- `responseDeadline` と `催促日` を分ける

Compass らしさは、「細かく管理させる」ことではなく、
「受け渡しを雑に回してもプロジェクトが進む」ことにある。
