## 詳細設計

### #1 担当者キャッシュ

- **問題**: TaskModal で毎回 `listProjectMembers()` API呼び出し → 「メンバー読み込み中...」表示
- **解決**: `useQuery` でキャッシュ。2回目以降は即座に表示
- **技術**: `@tanstack/react-query` （既にインストール済み）
- **キャッシュ戦略**: staleTime 5分、gcTime 10分、refetchOnWindowFocus: false
- **無効化**: メンバー追加・削除時に `invalidateQueries`

### #2 エラー報告

- **自動収集**: Sentry React SDK。500系APIエラーを自動送信。Error Boundary で未キャッチエラーもカバー
- **手動報告**: 画面右下にフィードバックボタン。種別選択（不具合/要望/その他）+ テキスト入力
- **コスト**: Sentry無料枠（5K events/月）で28人は十分
- **環境変数**: `VITE_SENTRY_DSN` （未設定時はgraceful degradation）

### #3 AI自動工程生成（将来）

- PJ作成後に「AIで工程を自動生成」ボタン
- 入力: プロジェクト種別、マイルストーン日程、クライアント情報
- AI: Gemini Flash で工程+タスクをJSON推論
- 出力: レビュー画面（bulk-importと同じUI） → 編集 → 一括保存
- UI原則2（AI出力にアクションボタン）に準拠

### #4 カレンダー双方向同期（将来）

- Phase 1: `startTime`/`endTime` フィールド追加。打合せは時刻必須
- Phase 2: Compass → Google Calendar Push同期（タスク保存時）
- Phase 3: Google Calendar → Compass Pull同期（Webhook）

### #5 ボール管理（将来）

- ステータス追加なし（Sena判断で見送り）
- 将来: カンバンビューをモバイルPWAとして実装
- スワイプで状態遷移（未着手 → 進行中 → 確認待ち → 完了）

### #6 AI課金制度（将来）

- **方針**: ベース料金に含む（制限付き）
- **理由**: 利用を躊躇させるとツールの価値が下がる
- **制限**: Small 30回/月、Standard 100回/月、Business 300回/月
- **超過時**: 「今月の利用上限に達しました」表示
- **サークル会員**: 現状の10回/日制限を維持

## 決定事項

- Senaの判断: ステータス追加は不要
- 実装順序: #1 → #2 → その他は後日
- Codex CLI で実装（指示書ベース）
